# Jibunkaigi Roadmap

## このロードマップの役割

この文書は、じぶん会議を今後どういう順番で設計・実装していくかを固定するためのものです。  
細かな実装手順書ではなく、中長期の進め方のコンパスとして使います。

重要なのは、早く要素を足すことではありません。  
何を先に固定し、何を後回しにするかの順番を誤らないことです。

## Phase 0 席を定義する

### 目的

じぶん会議が市場で取りに行く席を明確にし、今後の実装が「複数AIの掛け合いアプリ」に流れないようにする。

### 何を決める / 実装するか

- じぶん会議の本質を、固定人格ではなく内的傾向の凝集体験として定義する
- ユーザーに提供したい価値を、エンタメ性ではなく内的整理の深さとして言語化する
- 心の鏡、ジョー、他エージェント、ランダムモードをこの席の中で再配置する

### 到達条件

- プロダクトの判断基準として参照できる文書がある
- 何を目指さないかが明文化されている
- 以降の実装PRが、この席に照らして説明できる

### まだやらなくてよいこと

- 各エージェントの具体的なプロンプト調整
- UIでの見せ方の最適化
- パラメータの細かなチューニング

## Phase 1 共通OSを決める

### 目的

各エージェントや各モードの下にある共通内部力学を定め、場当たり的な個別実装を避ける。

### 何を決める / 実装するか

- 入力から発話、余韻更新までの最小フローを定義する
- field / reaction / stance / permission などの潜在カテゴリを固定する
- winner-take-all ではなく、競合・混合・抑制・慣性・偶発性を扱う前提を置く

### 到達条件

- どのエージェントにも共通する最小OSの図式がある
- 実装者がどこに新しい判断ロジックを置くべきか迷わない
- 以降のPRで、人格ごとの例外処理ではなく共通力学として説明できる

### まだやらなくてよいこと

- 完全なモジュール分割
- 全カテゴリの網羅
- 学習機構の高度化

## Phase 2 潜在層 / 顕在層を分ける

### 目的

何が本体で、何が表面化の窓かを切り分け、固定人格文が内部構造を支配しないようにする。

### 何を決める / 実装するか

- 潜在層を英語キー＋連続値で扱う前提を定義する
- 顕在層を短いテキスト片として再定義する
- 固定長文人格文を主役から外し、潜在状態の軽い読みに位置づけ直す

### 到達条件

- 内部状態と表面テキストが別物として扱われている
- 顕在表現の変更が、OSの根本設計変更を意味しない
- 人格の「設定文」より、潜在状態の推移が重視される

### まだやらなくてよいこと

- 全エージェントの表現差を完成させること
- 顕在層の文体を完璧に揃えること

## Phase 3 winner-take-all をやめて、競合と混合にする

### 目的

最も強い一要素だけを採用する構造をやめ、複数傾向の同時存在を内部構造として扱えるようにする。

### 何を決める / 実装するか

- 複数反応の競合、混合、抑制、慣性を扱う設計へ寄せる
- 発話前に単独勝者を決め切らない流れへ変える
- 少量の偶発性を、場を壊さない範囲で組み込む

### 到達条件

- 出力が単一ラベルの選定結果に見えなくなる
- 迷い・矛盾・揺れが、ノイズではなく構造として扱われる
- 実装判断で「誰が勝つか」より「どう混ざるか」が主語になる

### まだやらなくてよいこと

- 複雑な機械学習化
- 全状態の精密モデル化

## Phase 4 ジョーを新OSで再定義する

### 目的

ジョーを単なる前向き人格ではなく、新OS上で最も象徴的な役割を担う存在として再定義する。

### 何を決める / 実装するか

- ジョーがどの反応や姿勢を前に出しやすいかを定義する
- 「発火点」としての役割を、固定キャラ設定ではなく内部力学に接続する
- ユーザーの停滞やためらいに対し、どのような前進圧を与えるかを明確にする

### 到達条件

- ジョーの役割が、文体や勢いではなくOS上の位置で説明できる
- ジョー固有の価値が、他エージェントとの差分として見える
- 以降のプロンプト調整が、役割定義に沿って行える

### まだやらなくてよいこと

- ジョーだけを特別実装し切ること
- 他全員との完全な整合を一度に終わらせること

## Phase 5 心の鏡を“全体の重力”へ

### 目的

心の鏡を要約器から引き上げ、会議全体の重力と未解決点を映す存在へ移す。

### 何を決める / 実装するか

- 個別発話の総和ではなく、会議全体の重みを読む役割にする
- 何が未解決か、何が静かに支配しているかを返せる設計にする
- 閉じるためのまとめではなく、次に効く鏡面として再定義する

### 到達条件

- 心の鏡の出力が、単なる整理文ではなくなる
- ユーザーが「会議の重心」を受け取れる
- 他エージェントの平均では代替できない役割が明確になる

### まだやらなくてよいこと

- 完璧な総括品質
- すべての会話パターンへの最適化

## Phase 6 他エージェントを固定人格から“前に出やすい傾向”へ移す

### 目的

他エージェントも固定人格の箱から外し、内部傾向の前景化パターンとして扱えるようにする。

### 何を決める / 実装するか

- 各エージェントを「何をしがちか」「何を前に出しやすいか」で定義し直す
- 固定人格文の依存を弱め、潜在層との対応を強める
- 傾向同士の重なりや隣接を許容する

### 到達条件

- エージェントの違いが、キャラクター紹介文より内部傾向で説明できる
- 将来の再編や追加がしやすくなる
- 同じエージェントでも場に応じて出方が変わる前提が定着する

### まだやらなくてよいこと

- 名前や見た目の全面刷新
- すべての傾向ラベルの最終確定

## Phase 7 ランダムモードを場適合型にする

### 目的

ランダムモードを単純抽選から外し、その場に必要な傾きを選びやすくする。

### 何を決める / 実装するか

- 文脈適合を前提にした選出ロジックへ移す
- 必要だが普段勝ちにくい傾向を前に出せるようにする
- 偶発性は残しつつ、無秩序にはしない

### 到達条件

- ランダムモードに必然性が生まれる
- 同じ偏りだけが繰り返し出る状態を避けられる
- ユーザーが「今これが出てくるのは分かる」と感じられる

### まだやらなくてよいこと

- 高度な推薦最適化
- 長期履歴を使った複雑な制御

## Phase 8 最小リリース体験に削る

### 目的

ここまでで定めた思想と内部構造を保ったまま、最小のリリース体験へ削り込む。

### 何を決める / 実装するか

- ユーザーが最短で価値を感じる導線に絞る
- 必須でない演出や分岐を削る
- PR単位で安全に実装できる最小構成へ落とす

### 到達条件

- リリース候補として説明できる最小体験が定義されている
- 守るべき核を失わずに機能を絞れている
- PR-1以降の実装順が具体化されている

### まだやらなくてよいこと

- 全機能の同時完成
- 周辺機能の拡張
- 演出強化のためだけの追加実装

## フェーズ順序を守る理由

この順序は、思想→OS→層分離→競合混合→役割再定義→最小体験、という依存関係に基づいています。  
順番を飛ばして先に表現や機能を増やすと、あとで土台を変えたくなったときに全体が崩れます。

今後のPRは、このロードマップのどのPhaseに属するのかを明記したうえで進めるのが望ましいです。  
それにより、何を今決め、何をまだ決めないかが共有しやすくなります。
